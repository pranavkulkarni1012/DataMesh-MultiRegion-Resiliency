The Risk

Why Multi-Region Resiliency Now?

Today, a single us-east-1 outage takes down the entire Data Mesh platform — registration APIs, all three compute platforms, governance enforcement, and every consumer query.
⚠️

Precedent

us-east-1 experienced major outages in 2017, 2020, 2021, and 2023. This is not theoretical.

📈

Business Impact

All consumer analytics, regulatory reporting, and real-time dashboards become unavailable during an outage.

🔒

Compliance

Regulatory requirements increasingly mandate multi-region resilience for critical data infrastructure.

Current State

Architecture Today (us-east-1 only)

Single point of failure: Every component runs exclusively in us-east-1. A regional outage = total platform downtime.

Recommendation

Active-Passive Warm Standby

The right balance of resilience, cost, and operational simplicity.

What It Means

  • ▶ Secondary region (us-west-2) pre-provisioned
  • ▶ Data replication runs continuously
  • ▶ Infra kept warm at minimum capacity
  • ▶ Manual failover after confirmed outage

Why Not Active-Active?

  • ❌ 2x infrastructure cost
  • ❌ Dual-write consistency complexity
  • ❌ Split-brain risks
  • ❌ Not needed for DR use case

Why Not Pilot Light?

  • ❌ Cold-start latency too high
  • ❌ Snowflake table recreation is slow
  • ❌ Cannot meet 60-min RTO target
  • ❌ Starburst cluster boot is 10-15 min

Critical Challenge

Iceberg Tables — The Hard Problem

The Problem

Iceberg metadata files contain absolute S3 paths. After CRR to us-west-2, metadata still points to us-east-1 bucket. If east is down, tables are unreadable.

metadata.json (replicated to west):
"location": "s3://bucket-us-east-1/table"
"manifest-list": "s3://bucket-us-east-1/snap.avro"
// ^^^ Still points to EAST! Unresolvable if east is down
Also: S3 CRR has no ordering guarantee. Metadata may replicate before data files, causing FileNotFoundException on read.

Solutions (Ranked)

Option 1: MRAP (New Tables) RECOMMENDED

Write using Multi-Region Access Point URLs. Metadata contains region-agnostic paths. MRAP auto-routes to surviving region.

Option 2: register_table (Existing) FOR EXISTING

During failover, re-register tables in us-west-2 Glue Catalog with path substitution automation.

Option 3: S3 Tables (Future) LONG TERM

AWS managed Iceberg service with automatic metadata path rewriting and snapshot ordering.

Networking

S3 VPC Endpoints & Network Paths

S3 Gateway VPC Endpoints are strictly same-region. A gateway endpoint in us-west-2 cannot access an S3 bucket in us-east-1. S3 Access Points are also regional.

Cross-Region Access (Partial Failover)

When some producers stay in east and compute runs in west:

Recommendation: Do NOT engineer cross-region access paths. If a producer hasn't set up CRR to us-west-2, accept their data as unavailable during outage. The cost/complexity of cross-region VPC access outweighs the benefit.

Starburst: New Access Point Aliases on us-west-2 buckets
Snowflake: New Storage Integration + External Volume for west
Databricks: New External Locations pointing to west buckets

Consumer Notifications

Notification Channels — Multi-Region Strategy

Consumers are notified of new data availability through three channels, each with distinct DR implications.

SNS

SNS → SQS

Platform publishes to SNS. Consumer's SQS (in consumer's account) is subscribed. Consumer polls SQS.

Challenge: SNS topics are regional. Consumer SQS in us-east-1 is unreachable during outage.

DR Strategy: Dual-Publish

  • ▶ Create SNS topic in us-west-2
  • ▶ Platform publishes to both SNS topics
  • ▶ Consumer creates SQS in us-west-2
  • ▶ Consumer app polls both queues
  • ▶ Implement idempotent processing

KFK

On-Prem Kafka

Platform Lambda/ECS pushes messages to on-prem Kafka via Direct Connect / VPN. Consumer apps consume from Kafka.

BLOCKER: us-west-2 must have Direct Connect / VPN to on-prem. If DX only terminates in us-east-1, Kafka is unreachable during outage.

DR Strategy

  • ▶ Provision DX/VPN from us-west-2
  • ▶ Configure us-west-2 Lambda with Kafka brokers
  • ▶ Validate firewall rules, ACLs, TLS
  • ▶ No consumer-side changes (Kafka is on-prem)

Consumer polls REST API for new notifications. API backed by ECS/Lambda + DynamoDB.

Simplest DR: Already covered by platform API failover (Route53 + DynamoDB Global Tables).

Recommendation: Encourage consumers to prefer the Notification Polling API where possible — it has the simplest DR story. For SNS-SQS consumers, implement dual-publish for zero failover action. For Kafka, resolve Direct Connect from us-west-2 as a prerequisite (lead time: 4-8 weeks).

B
Partial — Critical Flip, Non-Critical Stay

Critical producers have CRR; non-critical do not. Platform runs in us-west-2.

Critical Datasets Non-Critical Datasets Cross-tier Joins
Recommendation: Implement tiered resiliency SLA. Tier 1 (Critical) = mandatory CRR. Tier 3 = accepted downtime.
C
Producer Refuses to Flip in DR Test

Platform & compute in us-west-2, but Producer X stays in us-east-1 during resilience test.

Starburst: UNAVAILABLE (gateway EP is same-region) Snowflake: Possible (cross-region via managed network)
Mandate: Critical producers must contractually participate in DR tests. Escalate refusals to senior management.
D
Platform Flips, Producers Stay

Platform runs in west, but east is up. Producers continue writing to east. CRR feeds west.

All APIs All Compute Data: 15-min stale

This is the steady-state during any failover. The 15-minute staleness is the accepted RPO.

E
Immuta-Only Region Failure

Immuta's us-east-1 deployment fails, but AWS us-east-1 is otherwise healthy. Starburst queries fail immediately; Snowflake & Databricks continue with last-synced policies.

Starburst: Queries FAIL (runtime dependency) Snowflake: Works (stale policies OK) Databricks: Works (stale policies OK)
Action Required: Work with Immuta/Starburst to implement a policy cache with configurable staleness window (e.g., serve cached policies for up to 1 hour if Immuta API is unreachable).
