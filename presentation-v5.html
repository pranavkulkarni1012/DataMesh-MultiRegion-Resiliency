Critical Discussion Points — Dual-Region Gaps

Architectural changes requiring detailed analysis: registration, API enhancements, notifications, and Iceberg table sync
3 / 9
1
Producer Registration — Dual-Region Schema Creation
Today: Schemas (Glue, Snowflake, DynamoDB) created in east1 only. Need: Every data product / data store creation must trigger schemas on both regions. Existing products need one-time backfill to west2.
DISCUSSION POINTS
  • Sync dual-write vs async (east1 first, west2 via event)?
  • Backfill: inventory existing products, automated script vs manual?
  • Starburst Glue Catalog sync: Producer Notification API must update Glue Catalog in both regions so Starburst west2 reflects new partitions/data
2
DEX & Mesh API — Dual-Region Enhancement
Today: DEX & Mesh API target east1 only. Need: Enhance to register on both regions (similar to Swiss setup). Every create/update/delete must propagate to east1 + west2.
DISCUSSION POINTS
  • Which Swiss multi-region patterns can be reused vs need adaptation?
  • API contract: transparent dual-write internally, or new region parameter?
  • Region-aware config: distinct SF accounts, Glue endpoints, IAM roles per region
  • Rollback strategy if west2 fails after east1 succeeds
  • Schema drift detection & reconciliation between regions
3
Consumer Notification — Dual-Region SNS Publishing
Today: Consumer Notification Lambda publishes to east1 SNS only. Need: Dual-publish to both east1 + west2 SNS. Consumers not doing region failover must still receive notifications.
DISCUSSION POINTS
  • Lambda code change to dual-publish on every invocation
  • DR (east1 down): consumers with east1-only SQS lose notifications until failover
  • Cross-account SNS → SQS subscriptions needed in west2 (new IAM/SQS policies)
  • Publish failure: if one region fails — retry, DLQ, or accept partial?
4
Iceberg Table Updates on US-WEST-2 After Replication
Gap: After CRR replicates Iceberg files to west2, compute platforms don't know about new data. Replicated metadata.json has absolute east1 S3 paths. Need: Refresh/re-register west2 Iceberg tables after CRR.
DISCUSSION POINTS
  • Event-driven: S3 event on west2 (metadata.json arrival) → Lambda → table refresh
  • Path problem: need register_table with west2 paths or path substitution layer
  • Snowflake: ALTER ICEBERG TABLE REFRESH or re-register after CRR
  • Ordering risk: metadata may replicate before data files → need staleness buffer
  • Total latency: CRR (~15 min SLA) + refresh processing = west2 data staleness
Action Required: Each area needs a detailed design doc with sequence diagrams and team ownership before implementation. Backfill of existing products to west2 (Topic 1) is a prerequisite for DR readiness.

Operational Process
1
Manual Setup with Snowflake SRE
Private Listing creation is not self-service — requires coordination with Snowflake SRE team to configure provider & consumer accounts
2
New Schema → SRE Engagement
When a new data product schema is created, must engage Snowflake SRE to add the schema to the Private Listing. Not automated today.
3
Replication Activation
Snowflake handles data replication cross-region via SnowGrid — consumer account in west2 receives a shared database (read-only)
!
Key Limitations
  • Snowflake-only: Starburst & Databricks consumers cannot access non-flipped producer data during DR
  • Manual dependency: Every new schema addition requires SRE involvement — risk of delays
  • East1 dependency: If east1 has full outage, Private Listing replication stops — non-flipped data becomes stale
  • Unmanaged Iceberg validation: Must validate with Snowflake support (BCR-1528)
?
Open Discussion Points
MONITORING & FREQUENCY
  • Replication lag monitoring: how to track Private Listing sync freshness?
  • Alerting: what SLA for data staleness is acceptable?
  • Replication frequency: near real-time, hourly, or on-demand refresh?
  • Dashboard/observability for Private Listing health & status
NEW SCHEMA ONBOARDING
  • Can the SRE process be streamlined or partially automated?
  • Lead time for SRE to add a new schema to Private Listing?
  • Who owns the request workflow — producer team, platform team, or both?
Impact Summary: Private Listing is a Snowflake-only capability. During DR, Starburst & Databricks consumers lose access to non-flipped producer data entirely. This trade-off must be accepted or mitigated with producer S3 CRR onboarding.

DR Flip Sequence

Step-by-step failover from us-east-1 to us-west-2 during a sustained regional outage
7 / 9
Failover Steps
1
Assess & Approve
Confirm sustained us-east-1 outage via AWS Health Dashboard. Check S3 CRR replication status. Get VP Engineering approval.
5 minutes
2
Activate Platform APIs
Route53 failover → west2 ALB. DynamoDB west2 becomes write region. Scale up ECS services. Verify Lambda functions.
5-10 minutes
3
Activate Iceberg Tables
Validate latest metadata.json in west2 S3. Run register_table in west2 Glue. Verify referenced data files exist (staleness buffer).
10-15 minutes
4
Activate Compute Platforms
Snowflake: Run external table recreation automation. Starburst: Scale up west2 cluster. Databricks: Activate workspace. Immuta: Verify connectivity.
10-20 minutes (parallel)
5
Activate Notifications
Verify SNS west2 receiving messages. Kafka: on-prem, no impact. Verify Polling API reachable via CNAME.
5 minutes (parallel with Step 4)
6
Producers Flip
Reverse CRR (west2→east1). Switch writes to west2 S3. Non-flipped: no change (Private Listing continues).
Coordinated with producers
7
Notify Consumers & Monitor
Status: Platform operational in west2. List available vs unavailable datasets. Continuous monitoring.
5 minutes
Failover Trigger
Trigger: Confirmed sustained us-east-1 outage (>15 minutes)
Authority: Platform Team Lead + VP Engineering
Type: Manual (with automated detection + alerting)
Failback: Manual, after us-east-1 stable for >2 hours
